## 1. Project Overview

**API-KPI** is a Go REST API that serves as the **backend middleware** for a sports data management and analysis platform. It sits between a **frontend application** (Next.js, running on `localhost:3000`) and an **analysis generator service** (Python, running on `localhost:9090`).

### Core Workflow
1. Users register/login via this API and receive a **JWT token**.
2. Authenticated users **upload CSV files** containing sports data (e.g., player performance metrics, game statistics).
3. Uploaded files are stored on **IPFS** via the **Pinata** pinning service, and metadata is persisted in **MongoDB**.
4. Users select one or more uploaded files and submit a **prompt** (natural language question).
5. This API forwards the file addresses + prompt to the **analysis generator** (`http://127.0.0.1:9090/analysis-gen`), which processes the CSV data and returns text responses and/or chart images (base64-encoded).
6. The API relays the analysis results back to the frontend.

## 4. API Routes

All routes are registered in `cmd/api/main/main.go` using `net/http.Handle()`. Every route is wrapped with a CORS middleware (`enableCORS`). The server listens on **`0.0.0.0:8080`**.

| Route                | Method | Auth Required | Min Permission | Handler                    | Description |
| -------------------- | ------ | ------------- | -------------- | -------------------------- | ----------- |
| `/register`          | POST   | No            | —              | `RegisterHandler`          | Create a new user account |
| `/login`             | POST   | No            | —              | `LoginHandler`             | Authenticate and receive JWT |
| `/upload`            | POST   | Yes (JWT)     | 0 (Pending)    | `UploadFileHandler`        | Upload CSV/file to IPFS |
| `/upload-picture`    | POST   | Yes (JWT)     | 0 (Pending)    | `UploadPictureHandler`     | Upload profile picture to IPFS |
| `/files`             | POST   | No            | —              | `GetFilesHandler`          | Get all files for a user |
| `/search-file`       | POST   | Yes (JWT)     | 0 (Pending)    | `SearchFilesHandler`       | Search files by filename/institution/writer |
| `/user-info`         | POST   | Yes (JWT)     | 0 (Pending)    | `UserInfoHandler`          | Get user profile info |
| `/users`             | POST   | Yes (JWT)     | 0 (Pending)    | `GetUsersHandler`          | List all users (basic info only) |
| `/pending-users`     | POST   | Yes (JWT)     | 7 (Admin)      | `GetPendingUsersHandler`   | List users pending approval |
| `/change-permission` | POST   | Yes (JWT)     | 7 (Admin)      | `ChangePermissionHandler`  | Change user permission level |
| `/analysis-gen`      | POST   | Yes (JWT)     | 0 (Pending)    | `AnalysisGenHandler`       | Generate analysis from CSV files |

```go
type User struct {
    Email          string   `json:"email" bson:"email"`

    Username       string   `json:"username" bson:"username"`
    Institution    string   `json:"institution" bson:"institution"`

    ReqAmount      int      `json:"reqamount" bson:"reqamount"`     // for amount-based permissions
    ID             string   `json:"id,omitempty" bson:"_id,omitempty"`
    Files          []File   `json:"files,omitempty" bson:"files,omitempty"`
    ProfilePicture string   `json:"profilePicture,omitempty" bson:"profilePicture,omitempty"`
    AnalysisImages []string `json:"analysisImages,omitempty" bson:"analysisImages,omitempty"`
}
```

## 7. Authentication & Authorization

### Authentication Flow
1. User sends `POST /login` with `{ "email": "...", "password": "..." }`.
2. Server finds user by email, compares bcrypt-hashed password.
3. On success, returns a **JWT token** (HS256, 1-hour expiry) with `Subject = username`.
4. Client sends the JWT in the `Authorization` header for subsequent requests (raw token, **no "Bearer " prefix**).

### Authorization (`UserAuthorized`)
- Defined in `cmd/api/handlers/auth.go`.
- Parses JWT from `Authorization` header, validates signature and expiry.
- Looks up user in MongoDB by the token's `Subject` claim (username).
- Compares user's `permission` field against the required minimum level.
- Returns `401 Unauthorized` if token invalid, `403 Forbidden` if insufficient permissions.

### JWT Key
- Stored in `.env` as `JWT_KEY`.
- Loaded in `cmd/api/handlers/register.go` via `os.Getenv("JWT_KEY")`.

## 8. External Integrations

### 8.1 Pinata (IPFS)
- **Purpose**: Decentralized file storage for uploaded CSVs and profile pictures.
- **API**: `https://api.pinata.cloud/pinning/pinFileToIPFS`
- **Gateway**: `https://scarlet-implicit-lobster-990.mypinata.cloud/ipfs/<hash>`
- **Auth**: API key + secret via `.env` (`API_KEY`, `API_SECRET`).
- **Upload limit**: 10MB (parsed via `ParseMultipartForm`).
- Profile pictures limited to 5MB, only JPEG/PNG/GIF.

### 8.2 Analysis Generator Service
- **URL**: `http://127.0.0.1:9090/analysis-gen` (POST)
- **Request payload**: `{ "fileAddresses": ["<ipfs_url>", ...], "prompt": "<user_question>" }`
- **Response**: JSON with optional `chart_base64` (base64 PNG image), `text_response`, and `message` fields.
- This is a **separate Python service** that this API proxies requests to.

### 8.3 SendGrid (Email)
- Sends welcome email on user registration.
- API key in `.env` as `SENDGRID_API_KEY`.
- From address: `felipe.stawinski@gmail.com`.

### 8.4 Ethereum Blockchain
- **Library**: `go-ethereum` v1.16.3
- **Contract bindings**: Generated via `abigen` from ABI/BIN files in `cmd/api/contract/`.
- **Config**: Contract address and private key stored in `.env` (`CONTRACT`, `PRIVATE_KEY`).
- Note: Blockchain interaction is available but the active handlers primarily use IPFS/MongoDB.

## 10. Coding Standards & Patterns

### General Patterns
- **No external HTTP router**: Uses Go's standard `net/http.Handle()` with `http.HandlerFunc` adapters.
- **Handler pattern**: Each handler is a standalone function in its own file under `cmd/api/handlers/`. Handlers parse request, validate, interact with DB, and write JSON response.
- **CORS**: Applied globally via `enableCORS` middleware in `main.go` (allows all origins `*`).
- **Error handling**: Uses `http.Error()` with appropriate status codes. Errors are also printed to stdout via `fmt.Println`.
- **JSON encoding/decoding**: Uses `encoding/json` with `json.NewDecoder` for request bodies and `json.NewEncoder` for responses.

### Naming Conventions
- **Files**: Lowercase with hyphens (e.g., `analysis-gen.go`, `upload-ipfs.go`, `change-permission.go`).
- **Packages**: Standard Go conventions — lowercase, single-word (`handlers`, `models`, `config`, `database`).
- **Handlers**: Exported functions ending in `Handler` (e.g., `UploadFileHandler`, `LoginHandler`).
- **Structs**: PascalCase with JSON/BSON tags in camelCase.

### Response Format
- All responses set `Content-Type: application/json`.
- Success responses use `json.NewEncoder(w).Encode(data)`.
- Error responses use `http.Error(w, message, statusCode)`.

### Database Access Pattern
Each handler:
1. Creates a new MongoDB client: `db := database.NewMongoDB(config.MongoURI)`
2. Gets the collection: `db.Database(database.DbName).Collection(database.CollectionName)`
3. Creates a context with timeout: `context.WithTimeout(context.Background(), 5*time.Second)`
4. Performs operations (FindOne, Find, UpdateOne, InsertOne)

### File Upload Pattern
1. Validate JWT authorization.
2. Parse multipart form (10MB limit).
3. Extract file via `r.FormFile("file")`.
4. Save JSON metadata to `json-files/<filename>.json`.
5. Upload file to Pinata IPFS, receive hash.
6. Store IPFS gateway URL in user's `files` array in MongoDB.

## 11. How to Run

### Prerequisites
- **Go** >= 1.24.1
- **MongoDB** running on `localhost:27017`

1. Create a `.env` file at the project root with the required variables (see Section 9).

## 12. Key Considerations for Development

- **Single collection design**: All data (users, files, analysis images) lives in the `users` collection. Files are embedded arrays in user documents.
- **No connection pooling**: A new MongoDB client is created per request. Consider refactoring to a shared client for production.
- **JWT token format**: Sent raw in `Authorization` header (no "Bearer " prefix).
- **Bilingual codebase**: Comments and some string literals are in Portuguese (PT-BR), but code identifiers are in English.
- **Console logging**: The codebase uses `fmt.Println` and `println` extensively for debugging. No structured logging framework is in use.
- **File IDs**: Auto-incremented integers per user (not globally unique). Determined by finding the max ID in the user's existing files array.
- **CORS**: Currently allows all origins (`*`). The `register.go` file also has a local `enableCORS` function that restricts to `localhost:3000` (unused in the main router).
